How-to Guide

  1. Set up a reposity

    2. git init should do the job early and push and update code often.

  2. Understand the problem

    3. Use Generally Useful Resources to find out more information about what your're facing. If you need, use paper, draw diagrams, bite pencils.

    If the problem seems too difficult, try to simplify it, find a special case that may be easier to design and implement. If it seems too easy, generalise, invent more complicated variations, perhaps try to combine with other problems.

  3. Implement

    4. Start with whatever language you know best or feel most confortable with. If the problem is easy, pick a language you've never used before or one that you don't know too well yet. Don't be afraid to write ugly or nonidiomatic code as long as understand what's going on. You can clear up the mess later.

  4. Test

    5. Write tests that check both the ordinary and the exceptional cases of your program. Compile/interpret/eval whatever you've written and run the tests. Find bugs, fix them and repeat. Again, commit often lest your progress disappear into the abyss.

  5. Reimplement

    6. Now that you have a reference implementation, you can go wild. Reimplement the task in different language this time, perhaps the one you've been meaning to learn since years ago. Try to achieve the same outcome using different techniques, use different idioms. Try to make the code as short as possible or the contrary.

  6. Rules
    1. No stealing of other's people work, only use them as references.
    2. Spin as much as you like, but remember to spin for the "initative."
    3. Breaks are allowed, Quitting is not.
    4. Share your work, no point in hoarding your genius.
  7. Generally Useful Resources
    1. Google
    2. Wikipedia
    3. Stack Overflow
    4. codeacademy.com
    5. github.com
    6. rosettacode.org

FAQ

  1. Can I make up my own challenges and add them to the list of challenges?

    2. Yes, as long as you email me the challenges, and after evaluating the them, I will add it to the list.

  2. How many challenges are there?

    3. As of now, I have stored about 100 programming challenges for the roulette wheel, but I will be updating the list over time.

  3. Will I be able to see the full list of challenges?

    4. Yes, the website actually derived from a list of challenges that were created by another user, and so will be posted at the bottom of the site as a reference.

  4. I've already done most of the challenges from the list, and the spinner results in the same number of challenges I have already completed.

    5. Actually, that was to be expected, that's why I adding another spinner for different languages. If you can do one thing in one language, you should be able to do it in another, or even complete the same challenge with a different technique.

  5. So, I guess until you update the list, I guess I will just have to keep on spinning.

    6. Yep, I guess so, but if you want more challenges please email me some ideas and maybe even donate if you've like. It definitely helps increase the list.

  6. Hey I'm a beginner programmer, should I be joining the roulette table with the big boys?

    7. Honestly, the challenges are geared towards any level, and it may take you more time than other people, but all the power to you if you want to dive into the deep end or jump into the fires of "programming."
